# 1. القرار الاستراتيجي المثبّت

المنتج النهائي للنظام ليس قرار BUY أو REJECT، وليس روبوت فلترة. المنتج النهائي هو دراسة استثمارية متعددة المسارات، مدعومة بالأدلة، قابلة للتحديث، وتشرح كيف يمكن الاستفادة من الفرصة عملياً مع تقليل المخاطر.

القرار المساعد (WATCH / REJECT / Score) يبقى طبقة داخلية لتنظيم الأولويات فقط، ولا يحل محل الدراسة الاستثمارية.

# 2. الرؤية والمهمة

## 2.1 الرؤية

بناء قسم بحث وتطوير واستثمار ذاتي يعمل باستمرار لاكتشاف الفرص التجارية في النرويج، وفهمها، وجمع أدلتها، وتصميم مسارات عملية للاستفادة منها دون افتراض أن الشراء هو الحل الوحيد.

## 2.2 المهمة

تحويل كل فرصة مؤهلة يكتشفها النظام إلى ملف استثماري حي يوضح الحقائق، الافتراضات، الأدلة، العملاء المحتملين، نماذج الإيراد، المخاطر، الاختبار الصغير، وأفضل خطوة تالية.

## 2.3 حدود الوعد

- لا يضمن الربح أو النجاح.
- لا يشتري ولا يزايد ولا يتواصل نيابة عن المستخدم تلقائياً.
- لا يحوّل الافتراضات إلى حقائق.
- لا يصدر قرار شراء نهائياً؛ بل يقدم أساساً منضبطاً للقرار البشري.

# 3. تعريف النجاح

يُعد النظام ناجحاً عندما يكتشف الفرص تلقائياً، ويختار المؤهل منها للتحليل، ثم ينتج لكل فرصة دراسة قابلة للتنفيذ تحتوي على أكثر من مسار للاستفادة، وأدلة سوقية واضحة، وخطة اختبار منخفضة التكلفة، وتُحدَّث عند ظهور معلومات جديدة.

- تشغيل تلقائي دون سؤال يدوي عن كل إعلان.
- فصل واضح بين الحقيقة والافتراض والمعلومة الناقصة.
- توليد ثلاثة مسارات استفادة قابلة للتنفيذ على الأقل عند ملاءمة الفرصة.
- إظهار مصدر كل دليل وتاريخه ودرجة موثوقيته.
- قياس استهلاك البحث الخارجي وتقييده.
- عدم تكرار التحليل دون سبب أو دليل جديد.

# 4. نطاق النظام

## 4.1 داخل النطاق

- مزادات الإفلاس والتصفية وفائض الشركات.
- مخزون المتاجر وتجهيزات المحلات وأثاث المكاتب.
- أصول سهلة التخزين والنقل وإعادة البيع أو الوساطة.
- فرص يمكن تحويلها إلى شراء، وساطة، شراكة، تقسيم، بيع مسبق، أو إدارة تصفية.
- مصادر مثل FINN.no وAuksjonen.no وKonkurs.app وKonkurskupp وBjarøy ومصادر عامة موثوقة.

## 4.2 خارج النطاق افتراضياً

- الشراء والمزايدة الآلية.
- السيارات والمعدات الثقيلة والدفعات المعقدة تقنياً، إلا بقرار صريح لاحق.
- التنبؤات غير المدعومة أو الأرقام المختلقة.
- التواصل الآلي مع البائعين أو المشترين دون موافقة المستخدم.

# 5. المبادئ المعمارية الحاكمة

| **المبدأ**            | **التطبيق**                                                      |
|-----------------------|------------------------------------------------------------------|
| **Evidence First**    | لا توجد خلاصة مهمة بلا دليل أو وسم واضح بأنها افتراض.            |
| **Living File**       | لكل فرصة ملف واحد دائم يتحدث بدلاً من إنشاء تقارير منفصلة متكررة. |
| **Multiple Paths**    | الشراء مسار واحد فقط من عدة مسارات ممكنة.                        |
| **Human-in-the-loop** | القرارات الملزمة تبقى بيد المستخدم.                              |
| **Cost-aware Search** | كل طلب بحث خارجي له ميزانية وسبب وسجل.                           |
| **Idempotency**       | إعادة التشغيل لا تكرر السجلات أو التحليلات بلا تغيير.            |
| **Auditability**      | كل نتيجة قابلة للتتبع إلى مصدرها ومدخلاتها وإصدار المحلل.        |
| **Modularity**        | كل محرك مستقل ويمكن اختباره واستبداله دون كسر بقية النظام.       |

# 6. المعمارية العليا للنظام

**Sources → Discovery → Normalize → Deduplicate → Qualification → Living Investment File → Evidence Tasks → Market Intelligence → Scenario Engine → Risk & Economics → Next Action → Monitoring & Update**

المبدأ المركزي: لا يذهب كل إعلان مباشرة إلى تحليل عميق. يمر أولاً بمرحلة تأهيل منخفضة التكلفة، ثم تُنشأ الدراسة الكاملة فقط عندما توجد إشارة واقعية إلى قيمة محتملة أو مسار استفادة قابل للفحص.

# 8. ملف الفرصة الاستثماري الحي

هذا الملف هو المنتج الأساسي للنظام ومصدر الحقيقة الوحيد للفرصة. لا يُنشأ تقرير جديد منفصل عند كل تحديث؛ بل تُحدّث الأقسام المتأثرة مع الاحتفاظ بتاريخ الإصدارات.

1. هوية الفرصة والمصدر والروابط والصور والتواريخ.
2. ملخص تنفيذي قابل للتحديث.
3. الحقائق المؤكدة مع مصادرها.
4. الافتراضات مع سببها وطريقة اختبارها.
5. المعلومات الناقصة والأسئلة المفتوحة.
6. الأصول أو البضائع الداخلة في الصفقة.
7. السوق والعملاء والمشترون المحتملون.
8. مسارات تحقيق الإيراد.
9. سيناريو الشراء.
10. سيناريو الوساطة.
11. سيناريو الشراكة.
12. سيناريو تقسيم الدفعة.
13. سيناريو البيع المسبق.
14. سيناريو إدارة التصفية مقابل عمولة.
15. الاقتصاديات: التكلفة الكلية، القيمة المحافظة، رأس المال، السيولة والمدة.
16. المخاطر وإجراءات التخفيف.
17. خطة اختبار صغيرة منخفضة التكلفة.
18. أفضل مسار حالي ولماذا.
19. الخطوة العملية التالية.
20. سجل التحديثات والأدلة الجديدة.

# 10. محرك الأدلة

لا يبدأ البحث بسلسلة استعلامات عامة مفتوحة. يبدأ بأسئلة محددة مشتقة من المعلومات الناقصة ومن السيناريوهات المقترحة.

1. استخراج الادعاءات التي تحتاج إثباتاً.
2. تحويل كل ادعاء إلى سؤال بحث محدد.
3. اختيار المصدر الأنسب والأقل تكلفة.
4. البحث مع استخدام Cache قبل أي طلب خارجي.
5. حفظ الدليل مع المصدر والتاريخ ودرجة الموثوقية.
6. تحديث القسم المتأثر فقط في ملف الفرصة.
7. إيقاف البحث عند بلوغ معيار كفاية الدليل أو ميزانية الطلبات.

## 10.1 درجات الأدلة

- A: مصدر رسمي أو وثيقة أصلية أو بيانات سوق مباشرة.
- B: مصدر تجاري موثوق أو أكثر من مصدر مستقل متوافق.
- C: إعلان مشابه أو مؤشر غير مكتمل يحتاج تحققاً.
- D: افتراض أو استنتاج داخلي؛ لا يُعرض كحقيقة.

# 11. محرك السيناريوهات

يولّد المسارات بناءً على طبيعة الأصل والسوق والقيود، وليس قائمة ثابتة تُنسخ لكل فرصة. كل مسار يجب أن يحتوي على: منطق الربح، العميل، المتطلبات، التكلفة، المدة، المخاطر، الأدلة الناقصة، وأول تجربة.

| **المسار**              | **الوصف**                                                      |
|-------------------------|----------------------------------------------------------------|
| **الشراء وإعادة البيع** | شراء الأصل وإعادة بيعه كدفعة أو بالتجزئة.                      |
| **الوساطة**             | ربط البائع بمشترٍ أو عدة مشترين مقابل عمولة.                    |
| **الشراكة**             | تقسيم التمويل والتخزين والتشغيل بين شركاء.                     |
| **تقسيم الدفعة**        | التفاوض على جزء محدد أعلى طلباً أو أسهل حركة.                   |
| **البيع المسبق**        | اختبار الطلب أو الحصول على تعهدات قبل الالتزام المالي.         |
| **إدارة التصفية**       | تولي التسويق والبيع مقابل نسبة دون تملك كامل المخزون.          |
| **إضافة قيمة**          | تنظيف، إصلاح، تجميع، تصوير، تغليف أو تصنيف لرفع القيمة.        |
| **إعادة التوجيه B2B**   | بيع الأصول لمتاجر أو شركات أو جمعيات بدلاً من المستهلك النهائي. |

# 12. الاقتصاديات والمخاطر

لا يُحسب الربح من سعر المزاد وسعر إعادة البيع فقط. يجب أن تشمل التكلفة الكلية: السعر، الضريبة، عمولة المنصة، النقل، الفك، التخزين، الفحص، الإصلاح، التصوير، التسويق، الوقت، واحتمال عدم البيع.

- سيناريو محافظ، مرجح، ومتزن عند توفر بيانات كافية.
- حد أقصى منطقي للالتزام المالي، وليس توصية مزايدة.
- زمن تحويل الأصل إلى نقد.
- نسبة رأس المال المعرض للخطر.
- نقطة الإلغاء: معلومة أو سعر يجعل المسار غير منطقي.
- اختبار حساسية لأهم متغيرين أو ثلاثة.

# 13. التحديث وإعادة التحليل

يعاد التحليل فقط عند وجود Change Trigger حقيقي، مثل تغير السعر، إضافة صور أو وصف، ظهور دليل سوق جديد، وصول رد من البائع، اكتشاف إعلان مرتبط، أو تغير حالة المزاد.

- عدم إعادة كتابة الملف كاملاً بلا تغيير.
- حساب بصمة للمدخلات والأدلة.
- إعادة تشغيل الوحدة المتأثرة فقط.
- حفظ الفرق بين الإصدار السابق والجديد.
- إرسال تنبيه فقط إذا تغيرت الخطوة التالية أو المخاطر أو جدوى أحد المسارات.

# 14. Knowledge Graph – مرحلة لاحقة

تُربط الكيانات التالية: البائع، الشركة، موقع الأصل، فئة البضاعة، الإعلان، المزاد، المشتري المحتمل، السعر المرجعي، والفرص السابقة. الهدف هو اكتشاف أن عدة إعلانات قد تمثل تصفية واحدة أو سلسلة قيمة واحدة.

لا تدخل هذه الوحدة في أول إصدار تنفيذي. تُبنى بعد نجاح ملف الفرصة الحي ومحرك السيناريوهات والأدلة.

# 15. إدارة Brave Search API والتكلفة

تنبيه تجاوز حد الاستهلاك لا يعني وحده توقف الخدمة، لكنه يثبت ضرورة وجود حوكمة للطلبات داخل النظام.

- Cache للاستعلام والنتائج مع مدة صلاحية مناسبة.
- منع الاستعلامات المتطابقة والمتقاربة داخل نفس الدورة.
- حد أقصى للطلبات لكل تشغيل ولكل فرصة ولكل يوم.
- عدم استخدام API الحقيقي في الاختبارات الآلية.
- تسجيل query_id والسبب والتكلفة والنتيجة.
- استخدام البحث الخارجي فقط عندما توجد فجوة معرفية محددة.
- إيقاف البحث عند كفاية الدليل وعدم البحث لمجرد زيادة عدد المصادر.

# 16. الأمن والحوكمة

- مفاتيح API في متغيرات البيئة فقط، ولا تُحفظ في GitHub.
- صلاحيات قراءة افتراضية؛ أي فعل خارجي يحتاج موافقة صريحة.
- سجل Audit غير قابل للتعديل لكل تشغيل وقرار وتحليل.
- تحديد مصدر كل قيمة وعدم ملء القيم المجهولة بتخمين.
- إخفاء البيانات الشخصية غير الضرورية.
- إمكانية تعطيل أي مصدر أو محرك دون تعطيل النظام كله.

# 18. نقطة البداية التنفيذية

نبدأ الآن بـ V2.5.0 فقط. لا نبني Knowledge Graph، ولا لوحة جديدة، ولا محرك تعلم قبل نجاح الملف الاستثماري الحي.

## 18.1 أول مهمة برمجية

تعريف Schema موحد لملف الفرصة الاستثماري الحي، ثم كتابة طبقة تنشئ الملف من بيانات فرصة موجودة، وتحدّثه دون تكرار، وتحفظ تاريخ التغييرات.

## 18.2 معايير قبول V2.5.0

- إنشاء ملف واحد ثابت لكل opportunity_id.
- حفظ الحقائق والافتراضات والمعلومات الناقصة منفصلة.
- دعم إضافة الأدلة دون حذف التاريخ السابق.
- حفظ analysis_version وchange_hash.
- عدم إنشاء نسخة جديدة إذا لم تتغير المدخلات.
- إخراج تقرير مقروء يحتوي الملخص والخطوة التالية.
- اختبارات تلقائية دون الاتصال بـ Brave أو أي API حقيقي.

# 19. ما لا نفعله قبل اكتمال V2.5.0
